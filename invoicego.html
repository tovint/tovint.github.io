Invoice2Go Design By Tovin Thomas

"After looking at the current app, please come up with 3 different approaches which encourage new users to send their first invoice within their first week. Create as many wireframes screens as needed to fully describe each approach.

Pick what you feel is the best approach of the 3 concepts and create a high fidelity (UX and UI) version which would be tested with a group of new users. Be prepared to talk about how you would determine if the test was successful and what production assets would be sent to the engineering team."

Design Approaches

Approach One

My first approach was to redesign the way invoice creation is being done now. Currently, you can create an invoice only after creating an account. Why not have users create their first invoice as an onboarding process? After downloading the app, users are presented with a welcome message asking if they want to set up their first invoice. I used encouraging messaging by mentioning that the process can be done in "3 simple steps".

If the user decides to proceed, I broke the process into 3 quick parts: "Add Clients", "Add Items", and "Attachments". The strategy behind this is to keep it simple. I want the user to be able to quickly enter information and move on to the next step. I also added a helpful indicator at the top for the user to know what part of the process he/she is at and how many more steps need to be completed (3 dots at top).

At the end, the user can review his/her invoice before choosing to send it or not. If the user does decide to send it, he/she will be directed to the screen that will ask him/her to sign up/sign in first. I believe this approach can encourage users to send their first invoice within the first week because they have already done most of the work. All that is left is for the user to create an account, using the existing process. Once finished, he/she can easily send out the invoice that was just created.

PROS:

  • Early user engagement
  • Even if the user doesn't sign up, he/she will have exposure to how easy the process is
  • Increase user acquisition
  • User has already set up his/her invoice, sending it should be easy (more likely to do it)

CONS:

  • Onboarding process becomes longer (potentially)
  • Not every user needs to send an invoice right away
  • From an engineering standpoint, how do you store all this information without an account assoicated with it?

Approach Two

My second approach was to use reminders. I looked at the scenario where a user downloads the app and creates an account, but does not need to send an invoice right at that moment. However, he/she may have upcoming dates in the week where the user will need to create an invoice. For this situation, I designed a reminder feature that is shown right after account creation.

The message allows the user to schedule reminders during the week to let him/her know that he/she needs to create an invoice. For example, lets say a contractor is meeting with a new client named Ashley on Wednesday, October 7. He/she can choose to set a reminder for the end of the day on Wednesday to send an invoice to Ashley. The reminder would be sent as a push notification that gets delivered on the set date and time.

I believe this approach adds additional incentive for the user to come back to the app and create his/her first invoice. Additionally, this feature can be beneficial for creating future invoices rather than just the first one. Not all users will need to send an invoice right after downloading the app, but having this system allows them to create reminders that works with their schedule. Ultimately, it’s important to create convenience for the users and give them the flexibility to use the app in a way that fits them best.

PROS:

  • Sending a reminder is easy
  • Schedule invoices around user's needs
  • Reminders get users back to the app
  • Can incorporate into existing reminder feature

CONS:

  • No guarantee that a reminder will lead to a user creating and sending the invoice

Approach Three

My third approach was using strategic and effective notifications. Rather than having the app sit idle hoping that the user will open it again, using push notifications can alert and call attention to the user.

If the user hasn’t opened the app all week, sending a notification can get the user back into the app (Wireframe screens 1 and 2).

In another situation, maybe a user is overwhelmed with the invoice creation process, which is what is preventing them from using the app again. To solve this, I designed a template section that the user is notified through a push notification. Allowing users to see pre-filled templates can help them understand how easy it really is to create an invoice. The templates can also be directly editable, allowing the users to replace the information with their own (Wireframes screens 3, 4, and 5). The inspiration for this came from being able to select your invoice theme when you create an account.

The last notification could be a promotional one that targets just the first week. Since free users get only 3 invoices, sending a notification saying, “Send your first invoice by the end of the week, and get an additional invoice!” could encourage the user to create and send an invoice right away. Although the user would get an additional invoice, the ultimate goal would have the user realize the simplicity of creating an invoice and hopefully upgrade to a paid account (Wireframe screen 6).

PROS:

  • Push notifications are cheaper than SMS
  • Notifications keep user engaged with the app even if they are not directly inside of it
  • Creating meaningful templates to could guide a user

CONS:

  • Notifications can be a nuisance
  • Timing notifications can be tricky
  • Notifications might go unseen

High Fidelity Design (Approach One)

For the high fidelity design, I went with approach one. I felt this was the most ambitious approach of the three. It would be interesting to test and see if this approach works succesfully. In terms of design elements, I chose to stick with the color theme of the currrent app. I sampled the green from the logo. For the form elements, I used a more modern style by having input fields be represented with single lines rather than square containers. All links or buttons are green (consistency). The overall design was made to be simple and efficient.

TESTING:

I would test this app using an A/B study by releasing two versions. The first version would be the new version with the invoice creation as the onboarding sequence, and the second version would the existing one. I would monitor using engagement with the new process and see how many people select to set up their first invoice. From there, I would track user clicks and see if they are reaching the end of the invoice creation process, or if they are canceling in between. If users complete the process and sign up for an account, I would then check and see how long it takes them to send their invoice. Ideally, if users send the invoice right away or within the next few days, the process was succesful. However, I would compare the results with the existing process and see if more invoices were sent out in the first week or not. If this new design proves to be succesful, I would also test out the placement of the process. Instead of using the invoice creation as an onboarding sequence, I would move it to after the user signs in. It would be interesting to see which placement users would react better too.
